01 · Core Idea

From rediscovering
to compounding

Most LLM + documents workflows look like RAG: upload files, retrieve chunks at query time, generate. The LLM rediscovers knowledge from scratch on every question. Nothing accumulates. The LLM Wiki pattern flips this: knowledge is compiled once and kept current.

The old way

RAG · Retrieval

  • ✕LLM rediscovers knowledge from scratch on every query
  • ✕Synthesis must be re-derived each time
  • ✕No accumulation across conversations
  • ✕Contradictions surface and re-surface
  • ✕Explorations vanish into chat history
The new way

LLM Wiki · Compile

  • ✓LLM reads each new source and integrates it into a persistent wiki
  • ✓Cross-references are already there
  • ✓Contradictions are flagged at ingest time
  • ✓Synthesis reflects everything you've read
  • ✓Explorations get filed back as new pages
The Key Insight

The wiki is a persistent, compounding artifact.
It gets richer with every source you add
and every question you ask.

Obsidian = IDE → LLM = Programmer → Wiki = Codebase
02 · Architecture

Three layers,
one disciplined LLM

Each layer has a clear owner. Raw sources are immutable truth. The wiki is LLM-owned and constantly rewritten. The schema is co-evolved between you and the LLM.

Layer 01

Raw Sources

immutable

Your curated collection of source documents. Articles, papers, images, data files, transcripts. The LLM reads from them but never modifies them. This is your source of truth.

▸ raw/articles/2026-04-llm-pattern.md
▸ raw/papers/attention-is-all-you-need.pdf
▸ raw/assets/fig-01.png
▸ raw/notes/2026-04-02-call.md
Layer 02 · core

The Wiki

LLM-owned

A directory of LLM-generated markdown files. Summaries, entity pages, concept pages, comparisons, an overview, a synthesis. You read it; the LLM writes and maintains it — creating pages, updating cross-references, flagging contradictions.

▸ wiki/index.md
▸ wiki/log.md
▸ wiki/entities/llm.md
▸ wiki/concepts/rag-vs-compile.md
▸ wiki/synthesis/overview.md
Layer 03

The Schema

co-evolved

A document (CLAUDE.md or AGENTS.md) that tells the LLM how the wiki is structured, what conventions to follow, and what workflows to run on ingest, query, and lint. This is what turns a chatbot into a disciplined maintainer.

▸ CLAUDE.md
▸ defines page templates
▸ defines ingest workflow
▸ defines lint checklist
▸ evolves with your domain

03 · Operations

Three operations,
one compounding loop

Ingest feeds the wiki. Query reads from it. Lint keeps it healthy. Together they form a loop where every action makes the next one cheaper.

Op · 01

Ingest

Read source → discuss → write summary → update index → update entity/concept pages → append log

Op · 02

Query

Read index → find pages → synthesize with citations → optionally file answer back as a new page

Op · 03

Lint

Find contradictions, stale claims, orphans, missing pages, broken cross-refs → suggest questions & new sources

loop runs forever · the wiki keeps getting richer
1

Drop & discuss

Drop a source into raw/ and tell the LLM to ingest. Discuss key takeaways, then let it write a summary page, update the index, and touch 10–15 related pages across the wiki.

2

Ask & file

Ask questions against the wiki. The LLM reads the index, drills into pages, and synthesizes an answer with citations. A great comparison or analysis you asked for? File it back as a new page — explorations compound.

3

Health-check

Periodically ask the LLM to lint. It surfaces contradictions, stale claims superseded by newer sources, orphan pages with no inbound links, missing cross-references, and data gaps — then suggests what to investigate next.

04 · Navigation

Two files, two views

index.md is content-oriented — a catalog of every page. log.md is chronological — an append-only timeline of every action. Together they let the LLM navigate without embeddings.

06 · Build It With Claude Code

The Deep Dive

A complete, step-by-step implementation using Claude Code. From zero repo to a healthy, compounding knowledge base. Each step shows the exact commands and the markdown you'll write.

1
Setup

Install Claude Code & scaffold the repo

Claude Code is Anthropic's agentic CLI. Install it via npm, then authenticate. Create a fresh git repo — your wiki will live here as a versioned collection of markdown files. Git gives you history, branching, and collaboration for free.

terminal
# 1. Install Claude Code (requires Node 18+)
npm install -g @anthropic-ai/claude-code

# 2. Authenticate (opens browser, signs in with your Anthropic account)
claude

# 3. Scaffold the repo
mkdir my-wiki && cd my-wiki
git init

# 4. Create the three-layer structure
mkdir -p raw/articles raw/papers raw/notes raw/assets
mkdir -p wiki/entities wiki/concepts wiki/sources wiki/synthesis
touch wiki/index.md wiki/log.md

# 5. Launch Claude Code inside the repo
claude

Why git: Every ingest is a commit. If the LLM botches a refactor, git reset undoes it. The diff between commits shows exactly what an ingest changed across the wiki.

2
Schema

Write the CLAUDE.md schema

This is the single most important file. It's what turns Claude from a generic chatbot into a disciplined wiki maintainer. Define the directory structure, page templates, the three workflows (ingest / query / lint), and the conventions for index.md and log.md. Co-evolve this file over time as you learn what works.

CLAUDE.md
# LLM Wiki — Project Schema

## Layout
- raw/      immutable sources. NEVER modify.
- wiki/     LLM-owned. Create, update, delete freely.
  - entities/    people, products, orgs, papers
  - concepts/    ideas, methods, distinctions
  - sources/     one summary page per ingested source
  - synthesis/   overviews, comparisons, theses
- wiki/index.md  catalog. Update on every ingest.
- wiki/log.md    append-only timeline.

## Page template
Every wiki page starts with YAML frontmatter:
---
type: entity | concept | source | synthesis
title: ...
created: YYYY-MM-DD
updated: YYYY-MM-DD
sources: [[[src-...]], ...]
---

## Workflows

ingest <path-to-source>:
  1. Read the source fully.
  2. Discuss key takeaways with the user.
  3. Write wiki/sources/src-<date>-<slug>.md.
  4. Update every relevant entity / concept page.
  5. Create new pages for entities/concepts not yet in the wiki.
  6. Flag contradictions with existing pages explicitly.
  7. Update wiki/index.md.
  8. Append entry to wiki/log.md using prefix:
     ## [YYYY-MM-DD] ingest | <title>

query <question>:
  1. Read wiki/index.md first.
  2. Drill into relevant pages.
  3. Synthesize answer with [[citations]].
  4. If the answer is non-trivial, ask the user whether to file it
     as a new page under wiki/synthesis/.
  5. Append ## [YYYY-MM-DD] query | <question> to log.

lint:
  1. Scan for contradictions between pages.
  2. Find stale claims superseded by newer sources.
  3. List orphan pages (no inbound links).
  4. List important concepts mentioned but without their own page.
  5. Suggest 3–5 new questions to investigate or sources to find.
  6. Append ## [YYYY-MM-DD] lint to log.

## Conventions
- Use [[wikilinks]] for all cross-references.
- One source per file in wiki/sources/.
- Be terse. The wiki is for skimming, not reading end-to-end.
- When in doubt, ask the user before deleting a page.
3
First ingest

Drop a source & ingest it

Save an article, paper, or transcript into raw/. Then in Claude Code, simply say "ingest". Claude reads the source, summarizes it, updates every related page, and logs the action. Watch the diff in your editor — a single ingest can touch 10–15 files.

terminal · claude REPL
# Drop a source into raw/
curl -sL https://example.com/llm-pattern.md > raw/articles/2026-04-llm-pattern.md

# In Claude Code's REPL:
> ingest raw/articles/2026-04-llm-pattern.md

# Claude will:
#   1. read the file
#   2. summarize key takeaways aloud
#   3. write wiki/sources/src-2026-04-llm-pattern.md
#   4. create wiki/entities/llm.md if it doesn't exist
#   5. update wiki/concepts/rag-vs-compile.md
#   6. update wiki/index.md
#   7. append to wiki/log.md

# Review the diff:
git diff --stat
#  wiki/sources/src-2026-04-llm-pattern.md | 84 +++++++++++++++
#  wiki/entities/llm.md                  | 22 ++++-
#  wiki/concepts/rag-vs-compile.md       | 31 +++++++
#  wiki/index.md                         |  6 ++
#  wiki/log.md                           |  9 +++
#  5 files changed, 152 insertions(+), 4 deletions(-)

git add -A && git commit -m "ingest: llm-pattern article"
4
First query

Ask a question & file the answer

Ask any question. Claude reads index.md, drills into the relevant pages, and synthesizes an answer with citations. If the answer is non-trivial, ask Claude to file it back as a new synthesis page — your explorations compound.

terminal · claude REPL
> query: how does the LLM Wiki pattern differ from Vannevar Bush's Memex?

# Claude reads wiki/index.md, then:
#   - wiki/concepts/rag-vs-compile.md
#   - wiki/entities/memex.md
#   - wiki/sources/src-2026-04-llm-pattern.md

# Claude answers with citations:
"Both envision a personal, curated knowledge store with
 associative trails between documents [[memex]]. The key
 difference is maintenance: Bush couldn't solve who does the
 bookkeeping. The LLM Wiki pattern offloads that to the LLM
 [[rag-vs-compile]], making the wiki self-maintaining."

> great answer. File it as a synthesis page.

# Claude writes wiki/synthesis/memex-vs-llm-wiki.md
# and links it from index.md and the entities it cites.

git add -A && git commit -m "query: memex vs llm wiki"
5
First lint

Health-check the wiki

After a few ingests, ask Claude to lint. It will scan the entire wiki and report contradictions, orphans, stale claims, and gaps. Lint output isn't just cleanup — it's a list of next investigations. Run it whenever the wiki feels unwieldy, or roughly every 10 ingests.

terminal · claude REPL
> lint

# Claude reports:
## Lint report · 2026-04-12

Contradictions (2)
- [[llm]] claims "context windows double yearly"
  vs src-2026-03-trends.md claims "growth has slowed"
  → suggest: reconcile and date-stamp both claims

Orphans (1)
- [[marp]] has no inbound links
  → suggest: link from [[output-formats]]

Missing pages (3)
- "embedding" mentioned in 4 pages, no own page
- "MCP" mentioned in 2 pages, no own page
- "graph view" mentioned in 3 pages, no own page

Suggested sources to find
- "As We May Think" — Vannevar Bush, 1945
- A recent paper on long-context retrieval failures

# Claude appends to log.md:
## [2026-04-12] lint
- flagged 2 contradictions, 1 orphan, 3 missing pages
- suggested 2 sources

git add -A && git commit -m "lint: 2026-04-12"
6
Iterate

Open in Obsidian & co-evolve the schema

Open the same folder in Obsidian. Browse the graph view, follow links, watch pages update in real time as Claude edits them. As you use the wiki, you'll discover friction — page templates that don't fit, workflows that need an extra step. Update CLAUDE.md, and the next session picks it up. The schema is alive.

terminal · claude REPL
# Open in Obsidian (set as vault)
obsidian .

# Recommended Obsidian plugins
#   - Dataview      (queries over YAML frontmatter)
#   - Marp          (slide decks from wiki content)
#   - Web Clipper   (browser → raw/articles/)

# When you discover friction, refine CLAUDE.md:
> Update CLAUDE.md: when ingesting a paper, also extract
  the bibliography into a "Cites" section in the source page.

# Claude edits CLAUDE.md, future ingests follow the new rule.
git add CLAUDE.md && git commit -m "schema: extract bibliographies on ingest"

# At scale (~100+ sources), add a search CLI:
> Add qmd for hybrid BM25+vector search over wiki/.

# Claude installs qmd, wires it into the query workflow,
# and updates CLAUDE.md to use it.

The pattern in one line: Obsidian is the IDE, the LLM is the programmer, the wiki is the codebase. Treat CLAUDE.md like a README that grows with the project.

07 · Tips & Tricks

Sharpen the workflow

Practical moves that make the wiki faster to feed, easier to browse, and resilient as it grows.

Obsidian Web Clipper

Browser extension that converts any article to markdown. Two-click pipeline from "interesting" to ingested source in raw/articles/.

Download images locally

Set Obsidian's "Attachment folder path" to raw/assets/. Bind "Download attachments for current file" to a hotkey. URLs break; local files don't.

Graph view is your map

Obsidian's graph view shows the wiki's shape: hubs, orphans, clusters. It's the fastest way to spot gaps and over-connected pages that should be split.

Marp for slide decks

Marp is markdown-based slides. Obsidian has a plugin. Ask Claude to turn a synthesis page into a deck — useful for sharing findings without leaving the wiki.

Dataview over frontmatter

If Claude adds YAML frontmatter (tags, dates, source counts) to every page, the Dataview plugin can render dynamic tables and lists. Free dashboards from your existing data.

It's just git

The wiki is a git repo of markdown files. Push to GitHub for backup. Branch before risky refactors. Merge when satisfied. Version history comes for free.

08 · Why It Works

The boring part is the point

The tedious part of maintaining a knowledge base isn't the reading or the thinking — it's the bookkeeping. Updating cross-references. Keeping summaries current. Noting when new data contradicts old claims. Maintaining consistency across dozens of pages.

Humans abandon wikis because the maintenance burden grows faster than the value. LLMs don't get bored, don't forget to update a cross-reference, and can touch 15 files in one pass. The wiki stays maintained because the cost of maintenance is near zero.

Human's job
  • · Curate sources
  • · Direct the analysis
  • · Ask good questions
  • · Think about what it all means
LLM's job
  • · Everything else
  • · Summarize
  • · Cross-reference
  • · File, lint, bookkeep

Lineage: The idea is related in spirit to Vannevar Bush's Memex (1945) — a personal, curated knowledge store with associative trails between documents. Bush's vision was closer to this than to what the web became: private, actively curated, with the connections between documents as valuable as the documents themselves. The part he couldn't solve was who does the maintenance. The LLM handles that.
